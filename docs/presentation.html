What is EmbedMatch?

  • A web platform to compare embedding models for text retrieval
  • Users upload documents, select models, and run benchmarks
  • Compares Azure OpenAI API models vs Open-Source models
  • Measures: retrieval quality, latency, and cost

Key question: Are free open-source models good enough, or do you need to pay for API-based ones?

Component Communication

1. Upload Flow

  1. Frontend → POST /uploads/
  2. Backend uploads file to Blob Storage
  3. Returns blob name to frontend

2. Start Benchmark

  1. Frontend → POST /experiments/
  2. Backend creates doc in Cosmos DB
  3. Backend enqueues job to Storage Queue
  4. Returns experiment ID

3. Run Benchmark (async)

  1. Function App dequeues message
  2. Downloads blob from Storage
  3. Generates queries via GPT-4o-mini
  4. Embeds docs with each model
  5. LLM-judges retrieval quality
  6. Writes results to Cosmos DB

4. View Results

  1. Frontend polls GET /experiments/{id}
  2. Backend reads from Cosmos DB
  3. Renders leaderboard when complete

Dual-Mode Embedding Engine

API Models

  • Azure OpenAI SDK call
  • ~6 ms per document
  • Pay per token
  • Models hosted by Microsoft
  • Requires network round-trip

Local Models

  • fastembed / ONNX Runtime
  • ~250 ms per document (CPU)
  • Free (compute only)
  • Models downloaded to Function App
  • Single-model eviction to fit B1 RAM

Asynchronous Processing

Why Queue-Based?

  • Benchmarks take 30–120s per model
  • HTTP timeout would kill the request
  • Queue decouples web tier from compute

Flow

  1. Backend enqueues experiment ID
  2. Function App dequeues & processes
  3. Results written to Cosmos DB
  4. Frontend polls until complete

Fault Tolerance

  • Visibility timeout: 10 min (covers model download on first run)
  • Max dequeue count: 3, then → poison queue
  • Function auto-retries on transient failures

Poison Queue

  • Failed messages go to benchmark-jobs-poison
  • Can be re-enqueued or inspected manually
  • Prevents infinite retry loops

Key Concepts

  • Kudu — Azure's SCM/deployment engine running alongside every App Service; exposes REST API for zip deploy, logs, SSH
  • Oryx — Azure's build system that auto-detects language, runs pip install / npm install from the zip during deploy
  • OneDeploy — Simpler alternative: pushes pre-built artifacts directly (no build step on server)

CI/CD Configuration Details

Backend Deploy

  • Zips backend/ with requirements.txt
  • Kudu receives zip → Oryx detects Python → runs pip install
  • Startup: uvicorn src.main:app

Frontend Deploy

  • pnpm build on CI runner
  • Zips dist/ folder (static HTML/JS/CSS)
  • OneDeploy pushes pre-built files
  • Startup: npx serve . -s

Function App Deploy

  • pip install -t fn_pkgs on CI runner (bundles 285 MB of packages)
  • Copies into .python_packages/lib/site-packages/
  • Zips entire bundle → Kudu zip deploy
  • No Oryx build — packages are pre-installed

Authentication

  • Publish Profiles stored as GitHub Secrets
  • Credentials extracted via XML parsing in CI
  • Passed to Kudu API as HTTP Basic Auth

Function App Deployment Challenge

Problem

  • fastembed + ONNX Runtime = 285 MB of packages
  • Remote build (pip install) unreliable on App Service Plans
  • First attempt with PyTorch: 3.3 GB zip — too large!

Solution

  • Switched to fastembed (ONNX Runtime) — 10x smaller than PyTorch
  • Bundle packages directly in the zip at .python_packages/
  • CI pipeline runs pip install -t then zips everything

Model weights (~80-420 MB) are downloaded on first invocation and cached at /home/hf_cache

Why Cosmos? Flexible schema, global distribution ready, serverless pricing tier available

Security Measures

  • IP Whitelisting — both App Services have firewall rules restricting access
  • Cosmos DB Firewall — only backend App Service outbound IPs allowed
  • Publish Profiles — credentials stored as GitHub Secrets
  • Connection Strings — stored in Azure App Settings (not in code)
  • CORS — backend only accepts requests from the frontend domain
  • File Validation — upload endpoint restricts to CSV/JSON/TXT, max 50 MB

Student License Drawbacks

Region & Resource Restrictions

  • Many regions locked (US, Asia, Australia, Canada, Brazil, etc.)
  • Some resource types unavailable in allowed regions (e.g., no Linux Consumption Plan in Switzerland North)
  • Azure OpenAI quota per region — Switzerland North has zero chat model quota → had to deploy OpenAI in Sweden Central
  • Limited TPM (tokens per minute) per model — throttles large benchmarks
  • $100 credit cap — no pay-as-you-go scaling

No RBAC (Role-Based Access Control)

  • Cannot create Service Principals → no passwordless CI/CD (OIDC)
  • Cannot create Managed Identities → services authenticate via connection strings instead of identity
  • No access to Azure AD / Entra ID app registrations → cannot add user authentication
  • Cannot assign roles (e.g., "Storage Blob Data Contributor") → must use full connection strings with account keys
  • Cannot set up VNet integration or Private Endpoints → must rely on IP whitelisting

Summary

  • 6 Azure services working together via queues, REST APIs, and SDKs

  • Dual embedding engine — API-based and local ONNX models in one platform
